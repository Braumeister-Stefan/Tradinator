## Inputs

Tradinator reads two input files at pipeline startup: a universe definition and a master price series. A third file — the candidate registry — is maintained separately for human inspection.

| Term | Definition |
|---|---|
| **epic** | IG's unique instrument identifier (e.g. `IX.D.FTSE.DAILY.IP`) |
| **universe** | The set of instruments the system knows about and may trade |
| **candidate** | Any instrument registered in `universe_candidates.json`, regardless of validation state |
| **valid** | An instrument that has passed both validation tiers and is eligible for trading |

### Universe Candidates Registry

`data/input/universe_candidates.json` is the **human-inspection file** for all 30 candidate instruments. It is the authoritative registry for adding, removing, or inspecting candidates. Do not edit `universe.json` directly — only `universe_candidates.json` should be edited to add new candidates.

Each entry records two validation tiers:

**Tier 1 — Broker Recognition**

| T1 Status | Meaning |
|---|---|
| `UNTESTED` | Validation has never been attempted |
| `PASS` | Broker recognises the epic and `dealingEnabled=true` |
| `EPIC_NOT_RECOGNIZED` | `fetch_market_by_epic` raised an exception or returned no data |
| `DEALING_DISABLED` | Epic found but `dealingEnabled=false` in market metadata |
| `API_ERROR` | Any other exception during the T1 API call |

**Tier 2 — Data Availability**

| T2 Status | Meaning |
|---|---|
| `NEVER_TRIED` | T1 has not passed yet; no data fetch attempted |
| `PENDING_T2` | T1 passed; pipeline has not yet run a data fetch for this epic |
| `YES` | Most recent data fetch returned ≥1 bar |
| `NO` | Most recent data fetch returned zero bars |

An instrument is **valid** (`valid: true`) only when `T1=PASS` AND `T2=YES`. The `valid` flag is set by `discover_universe.py` (based on T1+T2 test) and continuously updated by `DataPipeline` (based on live fetch results).

Current state: 30 candidates registered; 5 previously verified (pending re-validation).

| Asset class | Count | Regions |
|---|---|---|
| Indices | 15 | UK, US, EU, APAC |
| Forex | 9 | Global |
| Commodities | 6 | Global |

### Observable Universe

`data/input/universe.json` is the **machine-read universe file** used by the pipeline. It contains only `valid: true` instruments. Do not edit this file manually — it is managed automatically by `discover_universe.py` and `DataPipeline`.

To populate or refresh the universe, run universe discovery (see [Running Universe Discovery](#running-universe-discovery)).

### Running Universe Discovery

Universe discovery validates all candidates in `universe_candidates.json` against the IG Demo API and updates both files. Run it explicitly:

```bash
# Via CLI flag (recommended — re-runs discover then starts the pipeline):
python main.py --discover

# Standalone (validates candidates without running the pipeline):
python data/input/discover_universe.py
```

Or set it as the default in `main.py`:
```python
config["run_discover"] = True
```

Discovery runs in two phases:
1. **Phase 1** — Validates all 30 candidates in `universe_candidates.json`. T1 checks broker recognition and dealing eligibility; T2 checks that price data is available.
2. **Phase 2** — If fewer than 20 candidates are valid, queries the IG search endpoint for additional markets and tests them.

Results are written to `universe_candidates.json` (all candidates, with validation details) and `universe.json` (valid instruments only).

### Time Series Data

`data/input/universe_series.xlsx` is the master price series, written by `DataPipeline` on every run. The pipeline uses **incremental fetching**: before each fetch, it checks the last stored timestamp for each instrument in the master file.

- **Cold start** (no stored data): fetches the last `lookback` bars (default: 5).
- **Incremental** (stored data exists): fetches only bars after the last stored timestamp, so only genuinely new bars are retrieved.

If a fetch returns zero bars for an instrument, that instrument is removed from `universe.json` and its T2 status is set to `NO` in `universe_candidates.json`. Re-run `discover_universe.py` to re-add it if data becomes available again.

| Sheet | Content |
|---|---|
| `mid_close` | Mid-price close — (bid + ask) / 2 |
| `bid_close` | Bid close |
| `mid_open` | Mid-price open — (bid + ask) / 2 |

**Historic series:** `data/input/historic_series/` accepts `.xlsx` files of the same three-sheet schema to backfill the master series. Files are validated on load; any file that fails a schema check is skipped with a warning. Ingestion runs automatically on every pipeline run.

Data merge precedence (highest to lowest): live fetched data > existing master values > historic file data.
